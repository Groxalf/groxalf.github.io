Configurando un esquema básico de un proyecto Node con Gulp, Karma, Eslint, Babel y Browserify

2016-04-22
Javascript/ Learning/ Software Craftmanship

Durante el último proyecto que he estado trabajando, hemos incluido una gran cantidad de herramientas de Node en el gulpfile para buildear, lanzar los tests y agilizar los despliegues.

Para explicar el objetivo de la integración de estas herramientas propondré un pequeño ejemplo configurando paso a paso el proyecto. Este pequeño spike trata de buildear un proyecto con Browserify y Babel. Lanzar los tests con Jasmine y Nodemon en una instancia del browser PhantomJS desde terminal para evitar el uso de navegadores instalados en el equipo y poder lanzar los tests de integración en entornos de despliegue. Y por último, tener la revisión de nuestro código en Javascript por parte de Eslint en un log que nos aparecerá por la terminal cuando no se ajuste al convenio que hemos preestablecido.

Después en el root folder de nuestro proyecto creamos un fichero app.js que usará express para proveernos un servidor al haremos peticiones en nuestro test.

App.js

Crearemos un directorio test que contendrá el fichero de test que vamos a usar en nuestro ejemplo llamado sampleTest.js

El test es muy simple. Hacemos una petición GET usando la librería superagent y comprobamos que nos devuelve para esa ruta la respuesta que hemos definido en nuestro app.js.

Creamos el gulpfile.js en el root de nuestro proyecto.

En el gulpfile creamos la tarea ‘build’ que se encargará de generar un file único con todo el código javascript necesario para funcionar por sí mismo. Browserify se encargará de agregar todas las dependencias que hallamos indicado en nuestros ficheros de forma recursiva. Babel será el encargado de convertir el código ES6 que hemos creado a ES5 para que sea compatible con un mayor número de navegadores.

En la tarea ‘test’, creamos una instancia con Nodemon de nuestro servidor (en este caso el fichero app.js), necesario para poder escuchar la request que tenemos en nuestro fichero de test. Y lanzamos el servidor de Karma, que será el encargado de ejecutar test que hemos indicado en su fichero de configuración, en la instancia del browser de PhantomJS desde consola. Una vez acaba la ejecución, llamamos al done() para indicar que la tarea ha concluido y apagamos la instancia de Nodemon para que el servidor no quede corriendo en segundo plano.

Una cosa más a tener en cuenta. El array que tenemos después del nombre de cada tarea en varias de ellas, como por ejemplo, ‘default’, [‘lint’, ‘build’, ‘test’], esto indica, que default, no se lanzará hasta que termine las tres que se han indicado en el array.

El fichero karma.conf.js con la configuración para lanzar nuestros tests en la instancia del navegador de PhantomJS

Creamos una configuración personalizada para lanzar PhantomJS en nuestro fichero de Karma para que al hacer la petición GET desde nuestro test, cambiando el parámetro –web-security, no nos de un problema de cross-origin y falle la petición de ejemplo.

Y por último las reglas que tendrá Eslint para revisarnos el código, en el fichero .eslintrc en el root folder de nuestro proyecto.

Aquí hemos indicado los entornos Node y Jasmine para evitar que diga que funciones o variables como “process” o “describe” no se han declarado. El nivel del lenguaje, en este caso ES6. Y establecemos las reglas recomendadas de Eslint y además impedimos que se pueda usar el “var” para declarar una variable, permitimos usar console.log en el código, el punto y coma cuando termina un statement y que solo se pueden usar comillas simples para las strings, objetos json, etc.
